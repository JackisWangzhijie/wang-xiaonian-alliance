# 智能座舱主交互系统 - 代码覆盖率测试配置

## 1. 概述

本文档定义了智能座舱主交互系统的代码覆盖率测试配置规范，确保软件质量符合汽车级安全标准。

### 1.1 适用范围
- 智能座舱主交互系统所有Java/Kotlin模块
- ASIL等级：ASIL-B ~ ASIL-D
- 符合ISO 26262功能安全标准

### 1.2 参考标准
- ISO 26262:2018 道路车辆功能安全
- ASPICE 3.1 软件工程过程
- AUTOSAR 软件开发指南

## 3. 覆盖率目标

### 3.1 通用代码覆盖率要求

| 覆盖率类型 | 目标值 | 最低可接受值 | 权重 |
|------------|--------|--------------|------|
| 语句覆盖率 (Line Coverage) | ≥80% | 75% | 30% |
| 分支覆盖率 (Branch Coverage) | ≥70% | 65% | 25% |
| 方法覆盖率 (Method Coverage) | ≥80% | 75% | 20% |
| 类覆盖率 (Class Coverage) | ≥90% | 85% | 15% |
| 指令覆盖率 (Instruction Coverage) | ≥75% | 70% | 10% |

### 3.2 安全相关代码覆盖率要求 (ASIL)

| ASIL等级 | 语句覆盖 | 分支覆盖 | 方法覆盖 | 类覆盖 | 备注 |
|----------|----------|----------|----------|--------|------|
| ASIL-A | 90% | 80% | 90% | 95% | 关键功能模块 |
| ASIL-B | 95% | 85% | 95% | 98% | 重要安全功能 |
| ASIL-C | 100% | 90% | 100% | 100% | 高安全等级 |
| ASIL-D | **100%** | **100%** | **100%** | **100%** | 最高安全等级，要求MC/DC |

### 3.3 修改条件/判定覆盖率 (MC/DC)

**ASIL-D级别强制要求 MC/DC 覆盖率达到100%**

MC/DC验证要求：
- 每个条件独立影响判定结果
- 最小测试用例数 = N + 1 (N为条件数)
- 使用专用工具：VectorCAST, LDRA, BTC EmbeddedTester

## 4. 排除规则

### 4.1 排除类/包列表
```
排除规则:
  - **/R.class
  - **/R$*.class
  - **/BuildConfig.class
  - **/Manifest.class
  - **/*Test*.class
  - **/*Mock*.class
  - **/databinding/**/
  - **/generated/**/
  - **/proto/**/
```

### 4.2 可豁免代码类型
1. **自动生成的代码**
   - Databinding 生成代码
   - Protocol Buffer 生成代码
   - Room数据库生成代码

2. **框架基础代码**
   - Android框架回调
   - 第三方库Wrapper

3. **ASIL级别豁免** (需质量经理审批)
   - 调试代码
   - 日志输出代码
   - UI布局相关纯展示代码

## 5. CI/CD集成配置

### 5.1 Jenkins Pipeline 集成
```groovy
stage('Code Coverage') {
    steps {
        sh './gradlew jacocoTestReport'
    }
    post {
        always {
            publishHTML([
                allowMissing: false,
                alwaysLinkToLastBuild: true,
                keepAll: true,
                reportDir: 'build/reports/jacoco/test/html',
                reportFiles: 'index.html',
                reportName: 'JaCoCo Coverage Report'
            ])
        }
    }
}
```

### 5.2 质量门禁 (Quality Gate)
```yaml
quality_gate:
  line_coverage:
    warning: 75%
    error: 70%
  branch_coverage:
    warning: 65%
    error: 60%
  method_coverage:
    warning: 75%
    error: 70%
  class_coverage:
    warning: 85%
    error: 80%
```

### 5.3 增量覆盖率检查
```groovy
// 仅检查本次修改的代码覆盖率
diffCoverage {
    violationRules {
        rule {
            limit {
                minimum = 0.8  // 新代码80%覆盖
            }
        }
    }
}
```

## 8. 审查与审计

### 8.1 覆盖率审查流程
1. **每日检查**: 开发团队自查
2. **迭代审查**: 迭代结束时的覆盖率评估
3. **里程碑审查**: 发布前的全面覆盖率审计

### 8.2 覆盖率不达标处理
1. **黄色警告** (达到最低值但未达目标):
   - 记录在案
   - 下一迭代优先补充测试

2. **红色阻断** (未达到最低值):
   - 代码合并阻断
   - 必须补充测试并重新验证
   - 质量经理审批豁免
