## 1. 概述

### 1.1 目的
本文档定义了智能座舱主交互系统的单元测试策略，确保软件模块的正确性、可靠性和可维护性。

### 1.2 范围
- **覆盖范围**：主交互系统所有业务逻辑层、数据访问层、服务层组件
- **不包含范围**：UI层（由UI自动化测试覆盖）、硬件集成测试

### 1.3 测试目标
| 指标 | 目标值 | 说明 |
|------|--------|------|
| 语句覆盖率 | ≥80% | 每行代码至少被执行一次 |
| 分支覆盖率 | ≥70% | 每个条件分支至少被覆盖一次 |
| 方法覆盖率 | 100% | 每个public方法至少一个测试用例 |
| 测试通过率 | 100% | 所有测试用例必须通过 |

## 3. 测试策略

### 3.1 测试金字塔

```
        /\
       /  \
      / E2E\          ← UI自动化测试（少量）
     /______\
    /        \
   / Integration\      ← 集成测试（中等）
  /______________\
 /                \
/    Unit Tests    \   ← 单元测试（大量） ← 本文档重点
/____________________\
```

### 3.2 测试类型

#### 3.2.1 正向测试（Happy Path）
- 验证正常业务流程的正确性
- 每个public方法至少一个正向测试用例

#### 3.2.2 边界值测试
- 输入参数的边界条件
- 数值范围的上下限
- 集合的空/单元素/多元素状态
- 字符串的空/空白/超长情况

#### 3.2.3 异常路径测试
- 非法参数处理（IllegalArgumentException）
- 空指针处理（NullPointerException）
- 业务异常处理（BusinessException）
- 超时/重试场景

#### 3.2.4 状态测试
- 对象状态转换验证
- 依赖服务状态变化影响

## 4. 测试设计规范

### 4.1 命名规范

#### 测试类命名
```
被测类名 + Test
示例：UserService → UserServiceTest
```

#### 测试方法命名（推荐Kotlin风格）
```kotlin
// 方式1：反引号描述性命名（Kotlin）
fun `should return user when user exists`()
fun `should throw exception when user not found`()

// 方式2：should_when_then 格式
fun shouldReturnUserWhenUserExists()
fun shouldThrowExceptionWhenUserNotFound()

// 方式3：given_when_then 格式
fun givenExistingUserWhenFindByIdThenReturnUser()
```

## 5. Mock策略

### 5.1 何时使用Mock

| 场景 | 策略 |
|------|------|
| 外部服务调用 | 必须Mock |
| 数据库访问 | 必须Mock |
| 当前时间获取 | 必须Mock（保证可重复性） |
| 随机数生成 | 必须Mock |
| 同一层级的依赖 | 建议Mock |
| 纯函数工具类 | 可选（通常不Mock） |

### 5.2 Mock验证原则
- **验证交互**：确认依赖方法被正确调用
- **验证参数**：确认传入参数符合预期
- **验证次数**：确认调用次数（`times()`, `never()`, `atLeast()`）

## 6. 覆盖率标准

### 6.1 覆盖率等级

```
┌─────────────────────────────────────────────────────────┐
│  等级1: 核心业务逻辑（必须达到）                          │
│  - 语句覆盖率 ≥ 90%                                       │
│  - 分支覆盖率 ≥ 80%                                       │
│  - 包括：支付、安全、用户认证等                           │
├─────────────────────────────────────────────────────────┤
│  等级2: 通用业务逻辑（目标达到）                          │
│  - 语句覆盖率 ≥ 80%                                       │
│  - 分支覆盖率 ≥ 70%                                       │
│  - 包括：常规CRUD、配置管理等                             │
├─────────────────────────────────────────────────────────┤
│  等级3: 数据类/VO（酌情处理）                             │
│  - 可由Lombok/Data类生成，重点测试业务方法                │
└─────────────────────────────────────────────────────────┘
```

## 9. 附录

### 9.1 常见测试模式

#### 参数化测试示例
```kotlin
@ParameterizedTest
@CsvSource(
    "0, 0, 0",      // 边界：全0
    "1, 1, 2",      // 正常：正数相加
    "-1, 1, 0",     // 边界：正负抵消
    "2147483647, 1, -2147483648"  // 边界：整数溢出
)
fun `should calculate sum correctly`(a: Int, b: Int, expected: Int) {
    assertEquals(expected, calculator.add(a, b))
}
```

#### 异常测试示例
```kotlin
@Test
fun `should throw exception when divide by zero`() {
    assertThrows<ArithmeticException> {
        calculator.divide(10, 0)
    }
}
```
