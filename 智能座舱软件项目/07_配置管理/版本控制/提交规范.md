## 1. 概述

本文档定义智能座舱主交互系统项目的Git提交信息规范，确保：
- 提交历史可读性和一致性
- 自动生成CHANGELOG
- ASPICE Level 3 可追溯性要求
- 与需求、缺陷、测试用例的关联

## 2. 提交信息格式

```
<type>(<scope>): <subject> [<traceability>]

<body>

<footer>
```

## 3. 元素详解

### 3.1 Type（类型）

类型必须是以下之一：

| 类型 | 描述 | 使用场景 | ASPICE对应 |
|------|------|----------|------------|
| **feat** | 新功能 | 实现新的软件需求 | SWE.1 |
| **fix** | 缺陷修复 | 修复发现的缺陷 | SWE.6 |
| **docs** | 文档 | 仅文档变更 | SUP.10 |
| **style** | 格式 | 不影响代码含义的变更（空格、格式化、分号等） | - |
| **refactor** | 重构 | 既不修复缺陷也不添加功能的代码变更 | - |
| **perf** | 性能优化 | 提升性能的代码变更 | SWE.3 |
| **test** | 测试 | 添加或修正测试 | SWE.4 |
| **chore** | 构建/工具 | 构建流程或辅助工具的变更 | - |
| **security** | 安全 | 安全相关的修复或改进 | SEC.1 |
| **revert** | 回退 | 撤销之前的提交 | - |

### 3.2 Scope（范围）

范围指定变更影响的模块或组件：

```
ai          - AI交互模块
appmarket   - 应用市场
msgcenter   - 消息中心
framework   - 系统框架
system      - Android系统层
common      - 公共组件
infra       - 基础设施/构建
config      - 配置管理
docs        - 文档
```

**示例**：
- `feat(ai): 添加语音唤醒功能 [REQ-045]`
- `fix(framework): 修复内存泄漏 [BUG-012]`

### 3.3 Subject（主题）

- 使用祈使句、现在时态："change" 而非 "changed" 或 "changes"
- 首字母小写
- 结尾不加句号
- 限制在50个字符以内

### 3.4 Traceability（可追溯性）

ASPICE Level 3 要求所有提交必须可追溯。使用以下格式：

| 标识 | 用途 | 示例 |
|------|------|------|
| REQ-XXX | 需求ID | `[REQ-001]` |
| BUG-XXX | 缺陷ID | `[BUG-042]` |
| TC-XXX | 测试用例ID | `[TC-101]` |
| CR-XXX | 变更请求ID | `[CR-003]` |
| SWA-XXX | 软件架构元素 | `[SWA-005]` |
| SEC-XXX | 安全需求 | `[SEC-001]` |

### 3.5 Body（正文）

- 详细描述变更的动机
- 与之前行为的对比
- 每行不超过72个字符
- 解释 **what** 和 **why**，而非 **how**

### 3.6 Footer（脚注）

用于引用相关issue、关闭issue或标注破坏性变更：

```
Closes: BUG-042
Related: REQ-001, REQ-002
Breaking Change: API接口变更
```

## 4. 完整示例

### 示例 1：功能开发
```
feat(ai): 实现语音助手唤醒功能 [REQ-045]

- 集成科大讯飞语音识别SDK
- 实现本地唤醒词检测
- 添加低功耗模式下的唤醒支持

Closes: REQ-045
Related: SWA-012
Test: TC-156, TC-157
```

### 示例 2：缺陷修复
```
fix(framework): 修复服务生命周期管理中的内存泄漏 [BUG-087]

Service绑定后未正确解除绑定导致Activity无法被GC回收。
在onDestroy()中添加unbindService()调用。

Closes: BUG-087
Regression: TC-201
```

### 示例 3：测试相关
```
test(appmarket): 添加应用下载安装单元测试 [TC-203]

- 添加DownloadManager的Mock测试
- 覆盖正常下载、断点续传、下载失败场景
- 测试覆盖率提升至85%

Closes: TC-203
```

### 示例 4：文档更新
```
docs(config): 更新配置管理计划 V1.1 [DOC-CM-001]

- 添加CI/CD流程说明
- 更新分支保护规则
- 补充基线管理检查清单
```

### 示例 5：破坏性变更
```
feat(api): 重构用户认证接口 [REQ-098]

新的认证流程支持OAuth 2.0和双因素认证。

BREAKING CHANGE: 认证API接口签名变更
- login(String username, String password) 已废弃
- 使用新的 authenticate(AuthRequest request) 方法

Migration Guide: docs/migration/auth-v2.md
```

## 5. 提交验证规则

### 5.1 自动化检查（CI/CD）
```bash
# commitlint 配置示例
module.exports = {
  extends: ['@commitlint/config-conventional'],
  rules: {
    'type-enum': [2, 'always', [
      'feat', 'fix', 'docs', 'style', 'refactor',
      'perf', 'test', 'chore', 'security', 'revert'
    ]],
    'scope-enum': [2, 'always', [
      'ai', 'appmarket', 'msgcenter', 'framework',
      'system', 'common', 'infra', 'config', 'docs'
    ]],
    'subject-empty': [2, 'never'],
    'subject-full-stop': [2, 'never', '.'],
    'subject-max-length': [2, 'always', 50],
    'body-max-line-length': [2, 'always', 72],
    'references-empty': [1, 'never']  // 警告级别的可追溯性检查
  }
};
```

### 5.2 Git Hook 配置

**commit-msg hook**（.git/hooks/commit-msg）：
```bash
#!/bin/sh

commit_msg_file=$1
commit_msg=$(cat $commit_msg_file)

# 检查基本格式
if ! echo "$commit_msg" | grep -qE "^(feat|fix|docs|style|refactor|perf|test|chore|security|revert)(\([a-z-]+\))?: .+"; then
    echo "❌ Error: Commit message format invalid"
    echo "Expected format: <type>(<scope>): <subject>"
    echo "Example: feat(ai): add voice recognition [REQ-001]"
    exit 1
fi

# 检查可追溯性（警告级别）
if ! echo "$commit_msg" | grep -qE "\[(REQ|BUG|TC|CR|SWA|SEC)-[0-9]+\]"; then
    echo "⚠️  Warning: Missing traceability reference [REQ-XXX], [BUG-XXX], etc."
fi

echo "✅ Commit message format valid"
exit 0
```
