## 1. 分支模型概述

本项目采用 **Git Flow** 分支模型，适用于ASPICE Level 3合规的汽车软件开发流程。

```
┌─────────────────────────────────────────────────────────────────┐
│                        分支策略概览                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   main (生产分支)                                                │
│    │                                                            │
│    ├── hotfix/  ──────┐                                         │
│    │                  ▼                                         │
│    ◄──────────────────●── 紧急修复                                │
│    │                                                            │
│    ├── release/  ─────┐                                         │
│    │                  ▼                                         │
│    ◄──────────────────●── 版本发布                                │
│    │                                                            │
│   develop (开发主线)                                              │
│    │                                                            │
│    ├── feature/  ─────┐                                         │
│    │                  ▼                                         │
│    ◄──────────────────●── 功能开发                                │
│    │                                                            │
│    ├── bugfix/  ──────┐                                         │
│    │                  ▼                                         │
│    ◄──────────────────●── 缺陷修复                                │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 2. 长期分支

### 2.1 main 分支
- **用途**：生产就绪代码，仅包含经过完整验证的版本
- **保护级别**：严格保护，禁止直接推送
- **合并来源**：仅接受来自 `release/*` 和 `hotfix/*` 分支的合并请求
- **命名规则**：`main`

### 2.2 develop 分支
- **用途**：开发集成主线，包含最新的开发进展
- **保护级别**：受保护，需要代码审查
- **合并来源**：`feature/*`, `bugfix/*` 分支
- **命名规则**：`develop`

## 3. 临时分支

### 3.1 feature 分支（功能分支）
- **命名规则**：`feature/<需求ID>-<简短描述>`
- **示例**：`feature/REQ-001-user-authentication`
- **来源**：`develop`
- **合并目标**：`develop`
- **生命周期**：从需求开始到功能完成并验证

**创建命令**：
```bash
git checkout -b feature/REQ-001-user-auth develop
```

### 3.2 bugfix 分支（缺陷修复分支）
- **命名规则**：`bugfix/<缺陷ID>-<简短描述>`
- **示例**：`bugfix/BUG-042-memory-leak-fix`
- **来源**：`develop`
- **合并目标**：`develop`
- **生命周期**：从缺陷确认到修复验证完成

### 3.3 release 分支（发布分支）
- **命名规则**：`release/v<版本号>`
- **示例**：`release/v1.2.0`
- **来源**：`develop`
- **合并目标**：`main` 和 `develop`
- **生命周期**：从代码冻结到正式发布

**版本号格式**：遵循语义化版本控制 (MAJOR.MINOR.PATCH)
- MAJOR：不兼容的API修改
- MINOR：向下兼容的功能新增
- PATCH：向下兼容的问题修复

### 3.4 hotfix 分支（热修复分支）
- **命名规则**：`hotfix/v<版本号>-<修复描述>`
- **示例**：`hotfix/v1.2.1-security-patch`
- **来源**：`main`
- **合并目标**：`main` 和 `develop`
- **生命周期**：紧急修复从创建到部署完成

## 4. 分支保护规则

### 4.1 main 分支保护
- [x] 禁止强制推送
- [x] 禁止直接推送，仅通过MR/PR合并
- [x] 至少2名审批者批准
- [x] 必须通过CI/CD流水线
- [x] 必须解决所有对话
- [x] 分支必须是最新的

### 4.2 develop 分支保护
- [x] 禁止强制推送
- [x] 至少1名审批者批准
- [x] 必须通过CI/CD流水线
- [x] 必须通过代码静态分析

### 4.3 release 分支保护
- [x] 禁止强制推送
- [x] 仅项目维护者可合并
- [x] 必须通过全部测试

## 5. 提交规范（Conventional Commits）

所有提交必须遵循 Conventional Commits 规范，详见 [提交规范文档](./提交规范.md)。

### 5.1 提交格式
```
<type>(<scope>): <subject>

<body>

<footer>
```

### 5.2 Type 类型
| 类型 | 用途 | ASPICE追溯 |
|------|------|------------|
| feat | 新功能 | 关联需求ID |
| fix | 缺陷修复 | 关联缺陷ID |
| docs | 文档变更 | 关联文档ID |
| style | 代码格式 | - |
| refactor | 代码重构 | - |
| test | 测试相关 | 关联测试用例ID |
| chore | 构建/工具 | - |
| perf | 性能优化 | 关联性能需求 |
| security | 安全修复 | 关联安全需求 |

## 6. 合并流程

### 6.1 Feature → Develop
1. 功能开发完成并通过单元测试
2. 创建 Merge Request 到 develop
3. 代码审查（至少1人）
4. CI/CD 流水线通过
5. 合并并删除 feature 分支

### 6.2 Develop → Release
1. 确定发布版本号
2. 从 develop 创建 release 分支
3. 进行集成测试和验收测试
4. 修复发现的问题
5. 创建 Merge Request 到 main

### 6.3 Release → Main
1. 版本验证完成
2. 更新版本号和CHANGELOG
3. 创建 Merge Request 到 main
4. 至少2人审批
5. 打版本标签
6. 合并到 main

## 7. ASPICE Level 3 合规要求

### 7.1 可追溯性
- 每个提交必须关联工作项（需求/缺陷/任务）
- 提交信息格式：`feat(模块): 描述 [REQ-001]`
- MR必须关联相关需求或变更请求

### 7.2 审计记录
- 保留完整的分支历史
- 禁止重写历史（rebase到main/develop）
- 所有合并必须通过MR/PR，保留审批记录

### 7.3 基线管理
- 每次发布创建基线标签：`baseline/v<版本号>`
- 基线创建后冻结代码
- 基线信息记录到基线历史文档

## 8. 工具配置

### 8.1 Git Hooks（推荐）
```bash
# commit-msg hook 检查提交信息格式
#!/bin/sh
commit_msg_file=$1
commit_msg=$(cat $commit_msg_file)

if ! echo "$commit_msg" | grep -qE "^(feat|fix|docs|style|refactor|test|chore|perf|security)(\(.+\))?: .+"; then
    echo "Error: Commit message does not follow Conventional Commits format"
    exit 1
fi
```

### 8.2 CI/CD 集成
- 分支命名规范检查
- 提交信息格式检查
- 自动代码审查分配
